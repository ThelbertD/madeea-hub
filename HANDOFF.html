3Things that are NOT automatic (important!)

Pushing to main only redeploys the frontend. These three must be done by hand:

a) Database changes (migrations)

SQL files live in supabase/migrations/. Apply a new one by pasting it into the Supabase → SQL Editor and running it (or npx supabase db push). Latest applied: 0012_shared_workspace.sql.

b) Edge functions (server code)

npx supabase login
npx supabase link --project-ref bglduxferbjmoeqzyypx
npx supabase functions deploy <name> --no-verify-jwt

All browser-called functions deploy with --no-verify-jwt and check auth inside the code — this is required so the browser's CORS preflight passes. Don't remove that flag.

c) Email sending (for invites & password resets)

Set custom SMTP in Supabase → Authentication → Emails (Brevo or Mailgun both work), and set the Site URL / Redirect URLs to the Vercel app so invite links land correctly.

4How security works (read this)

The real protection is in the database (RLS), not the buttons. Hiding a button in the UI is only cosmetic — every rule is also enforced by Postgres, so a tampered browser still can't do anything it shouldn't.
  • Every data table is scoped to the user's workspace: workspace_id = my_workspace().
  • Only admins can change roles or remove members (is_admin()), enforced server-side.
  • The invite function verifies the caller is an admin on the server and never trusts the browser for the workspace. The secret service key lives only inside the function.
There are two different "role" fields — don't confuse them:
  • profiles.role = a job title shown in the sidebar (e.g. "Elite EA"). Cosmetic — no permissions.
  • memberships.role = the real permission: admin or ea. This controls admin access.

6⚠ The one open task — trim the admin list

When everyone's separate accounts were merged into one workspace, their old admin roles were kept (so nobody got locked out). Right now 4 of 5 accounts are admins. Set the ones who shouldn't be admins back to EA:

-- see who is what
select u.email, m.role
from memberships m join auth.users u on u.id = m.user_id
order by m.role, u.email;

-- make everyone an EA, then re-promote only the real admin(s)
update memberships set role = 'ea';
update memberships set role = 'admin'
where user_id in (select id from auth.users where email in ('kyle@disruptorsmedia.com'));

Or just use the “Make EA” button in the Admin panel — role changes are reliable now.
